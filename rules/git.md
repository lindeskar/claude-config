# Git

> Full war stories behind these rules: `reference/git.md` in this repo (not auto-loaded).

## Commits

- Conventional commit messages, single line: `<type>[scope]: <description>`
- Append `!` after `<type>[scope]` for a BREAKING change — but `!` means the **released artifact** breaks for its consumers (removed CLI flag, changed API), not that the change needs careful operational sequencing. CI-template flips, config changes, and rollout steps are NOT breaking; express sequencing in the PR body/merge gate. A spurious `!` forces a major version bump of an artifact that didn't change.
- **Pick the commit/PR type by *consumer impact of the released artifact*, not by which paths changed.** In a repo that *publishes* something others consume — a **reusable workflow** (`on: workflow_call`) or **composite action** that other repos call, **or a served config artifact** — a behaviour change is `fix:`/`feat:` (it reaches consumers), even though it edits `.github/workflows/` or a `*.json`. `ci:`/`chore:` is only for *this repo's own* plumbing that no downstream consumes. The "published artifact" is not always code: **`annotell/renovate-config` is a release-please repo whose released artifact is the Renovate preset itself** (baked into the nginx-sidecar image every repo's bot consumes) — so any change to `default.json`/the sub-presets that alters bot behaviour (a new `packageRule`, `logLevelRemap`, cooldown tweak) is `fix:`, and `chore:` is reserved for genuinely non-shipping edits (README-only, repo tooling). Mislabeled `chore:` there ships to nobody until an unrelated `fix:`/`feat:` happens to cut the next release. Case: [annotell/renovate-config#74](https://github.com/annotell/renovate-config/pull/74) (a `logLevelRemap` behaviour change) was titled `chore:` and did **not** trigger a release — should have been `fix:`. Getting this wrong on a **release-please repo bites twice**: `ci` (and `chore`/`docs`/`refactor`/…) are **hidden types** — they render no changelog section, so the empty-changelog gate aborts the release PR and **no version is cut**; a consumer-facing fix mislabeled `ci:` lands on `master` but ships to nobody. Squash-merge uses the *PR title* as the commit, so the PR title is what's classified. Case: `gha-k8s-gitops#89` (create-github-app-token `app-id`→`client-id` fix in the reusable `copy-images`/`diff` workflows) should have been `fix:`, not `ci:` — it merged without cutting a release. See work-wiki memory `release-please-commit-type-visibility` for the hidden-type mechanics.
- Never amend commits
- Never force-push. When a task seems to require it (e.g. rebasing a pushed PR branch), use `git merge --no-ff <other-branch>` and push normally.
- Never use `git -C <path>` — always `cd` to the repo directory first
- Run `git add` and `git commit` as separate Bash tool calls — never chained (see `tooling.md`)
- **Scope `git commit` with an explicit pathspec (`git commit <paths> -m …`) whenever the repo might carry unrelated staged/uncommitted changes** — a bare `git commit` lands the *entire* index, not just what you `git add`ed. In a shared/personal notes repo mid-edit (e.g. `lindeskar/work` during a wiki cleanup), a bare commit once swept the user's in-progress work into an unrelated commit. When unsure, `git status --short` first; recover a bad bundle with `git reset --soft HEAD~1` (restores the pre-commit index, nothing lost).
- Never run `git commit` with `run_in_background: true` — 1Password signing needs an interactive prompt; backgrounded commits fail with `failed to fill whole buffer`. Same reason: a dispatched **subagent** (which runs backgrounded) can't make signed commits — in subagent-driven flows the controller makes the commits in the foreground. **In a headless/background *job* (not just `run_in_background` bash), even a foreground `git commit` can hit `failed to fill whole buffer` intermittently** — the 1Password prompt may surface for early commits then stop mid-session. For a throwaway/never-merged branch (a CI-trigger test PR), `git commit --no-gpg-sign` is an acceptable fallback; for anything that lands in history, retry once and otherwise ask the user to run the commit so the prompt appears. **The failure is also silent: in a bg job the commit can succeed with no error yet produce an *unsigned* commit — success ≠ signed. Verify after committing anything that lands in history, but use the *right* check: `git log --show-signature` / `git log --format=%G?` are unreliable here — they error with `gpg.ssh.allowedSignersFile needs to be configured` and report `No signature`/`N` *even for a correctly signed commit* (the local box has no allowed-signers file), so trusting them yields a false "Unverified" verdict. The reliable local check is the presence of the signature header: `git cat-file -p HEAD | grep '^gpgsig'` (`-----BEGIN SSH SIGNATURE-----` present ⇒ signed; GitHub will show it Verified). Only report Unverified when that header is genuinely absent.**
- **Always create commits with a local, signed `git commit` — never via the GitHub API.** API-created commits (`gh api --method PUT .../contents/...`) land **Unverified** (they never touch the 1Password SSH signing key). Don't script API commits to "save cloning" for multi-repo changes — clone each target locally (usually already under `~/Code/_kognic/`), branch, edit, commit signed.
- For multi-line commit messages: pass a multi-line string directly to `git commit -m` — never command substitution
- Match commit granularity to the task — bulk-adding new files is one commit, not one per directory. Split only when changes are incremental and benefit from independent review or rollback.
- `git pull --ff-only` / bare `git fetch` **fails on repos that move a floating major tag** (release-please action repos like `annotell/gha-golang` republish `v1` per release): `! [rejected] v1 -> v1 (would clobber existing tag)`. Resolve with `git fetch origin --tags --force`, then pull — `--force` only repoints the floating tag; immutable release tags are untouched. `--no-tags` does not help.

## Pull requests

- Always create PRs in draft mode with a description: `gh pr create --draft --title "..." --body "..."` — no exceptions, even throwaway/test PRs (a ready PR fires CODEOWNERS review-requests and notifies colleagues before CI finishes)
- **Set the final, valid Conventional-Commit title when you *create* the PR — never placeholder-then-rename.** The `annotell` org's "Validate PR title" ruleset checks only the `opened` event and does not re-run on rename, so a placeholder's failure sticks until a new commit or reopen. For titles containing `!` (which `gh pr create --title` mangles), set the title at creation via the API from a Write-created file — see `tooling.md`.
- **Before pushing a new commit to an open PR, check its review state** (`gh pr view <n> --json reviewDecision`) — a push to an `APPROVED` PR **dismisses the approval and forces a re-review**, which costs the reviewer a second pass. Don't push follow-up polish, nice-to-haves, or self-found refinements to an already-approved PR: put them in a separate PR, or flag the finding to the author and let them decide. Only push to an approved PR when the change is essential to *that* PR's correctness and can't reasonably wait — and say so. Editing the PR *body/title* does NOT dismiss approval; only new commits do.
- Add the `autoreview` label to PRs that only touch staging environments in `kognic-cd`, `k8s-platform-gitops`, or `k8s-platform-gitops-volcano` (`--label autoreview`). Never on PRs touching prod paths.
- **A throwaway change added only to trigger CI (e.g. a comment in a state dir to make a path-filtered terraform matrix run) must be reverted the moment CI confirms green — before the PR is marked ready — never left with a "remove before merge" marker.** Squash-merge folds every commit on the branch into the default branch, so the marker doesn't stop it landing; it just costs a follow-up cleanup PR. If the trigger must survive the review window, revert it in a second commit on the same branch rather than trusting anyone to strip it at merge.
- **When posting a code review as a PR comment**, wrap the body in a collapsible `<details><summary>Code review results</summary>` … `</details>` block (blank line after `<summary>` and before `</details>` so the markdown renders). Keep any `` `/anthropic-review-pr:review-pr` output: `` marker line *above* the `<details>`. Write the body with **Write** (reviews routinely contain `!`) and post with `gh pr comment <n> --repo <owner>/<repo> --body-file <path>`.

## PR/issue bodies and links

- PR body: lead with the **intent** (why / what it achieves), not a diff recap. Default to one sentence plus any tracking link; don't restate what the diff shows (no changed-file lists, counts, before/after strings). Backticks for identifiers/paths. Add a second sentence only when the reviewer needs context invisible in the diff; use a `## Summary` bullet section only for genuinely multi-concern changes. **Cut anything no reviewer will act on or ask about** — merge-sequencing notes, mechanism walk-throughs, file paths, "how it works", and hedged caveats belong in the code/commit or a comment when asked, not the body. If in doubt, shorter: the author will add detail if a reviewer needs it. (Recurring slip: 5-paragraph bodies with sequencing + config paths; the user trims them to 2-3 sentences.)
- PR body describes what changed, not what was intentionally omitted — don't enumerate non-changes unless the gap would surprise a reviewer.
- PR body readability: short, plain sentences; split run-ons; lead with the concrete cause a reader can picture; plain words over jargon. Separate distinct points (cause → fix → cross-reference) with blank lines, even in a 2-3 sentence body.
- **Cross-repo references — fully qualify, then format by surface.** Always resolve the ref to the repo the issue/PR actually lives in and verify that repo first — a bare or wrong-repo `#NNN` resolves into the wrong number space, often hitting an unrelated item (most plateng tracking issues live in `kognic-internal/devplat`, but not all — check). Then format by *where it renders*:
  - **GitHub PR/issue bodies and comments → paste the plain full URL** (`https://github.com/owner/repo/pull/NNN`), on its own or after a lead-in (`Tracked in …`, `Depends on …`). GitHub auto-renders it as a rich `owner/repo#NNN` reference with title/hovercard. Do **NOT** wrap it in `[text](url)` markdown — it's noisier and the user strips it. (Same-repo `#NNN` autolinks; a plain full URL is always safe cross-repo. For a file in another repo, paste the plain blob URL, not an inline-code path.)
  - **Chat replies to the user → use a clickable markdown link** `[owner/repo#NNN](https://github.com/owner/repo/pull/NNN)`. Here a bare URL or inline-code `owner/repo#NNN` does NOT render as a clickable rich link — the markdown link is required.
