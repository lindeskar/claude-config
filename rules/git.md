# Git

- Conventional commit messages, single line: `<type>[scope]: <description>`
- Append a `!` after `<type>[scope]` to indicate a BREAKING change
- `!` means the **released artifact** breaks for its consumers (removed CLI flag, changed API), not that the change needs careful operational sequencing. CI-template flips, config changes, and rollout steps are NOT breaking even when mis-timing them would cause failures — express sequencing in the PR body/merge gate, not the commit type. The squash title feeds release tooling; a spurious `!` forces a major version bump of an artifact that didn't change. (Learned on `gitlab-ci-volcano#182`.)
- Always create PRs in draft mode with a description: `gh pr create --draft --title "..." --body "..."`
- When a reply involves a PR (created, updated, or referenced), include a **clickable markdown link** to it, and always use the fully-qualified `<org>/<repo>#<n>` form as the link text: `[<org>/<repo>#<n>](https://github.com/<org>/<repo>/pull/<n>)` (e.g. `[annotell/terraform-devplat-volcano#455](https://github.com/annotell/terraform-devplat-volcano/pull/455)`). Don't cite a bare `#NNN` (not clickable in the terminal) and don't use a bare `#<n>` as the link text — the org/repo prefix makes it unambiguous which repo the PR lives in. Put the link in the final message so the user can jump straight to the PR.
- **Set the final, valid Conventional-Commit title when you *create* the PR — never create with a placeholder title and rename afterward.** An org ruleset's "Validate PR title" check (`annotell` repos with `require-semantic-pr-titles=true`) validates the title on the `opened` event and does **not** re-run on a rename (`edited` is not a default activity type and rulesets ignore `types:` filters). A placeholder title therefore gets validated and the failure sticks until a new commit or reopen. For titles containing `!` (which `gh pr create --title` mangles under interactive zsh), set the title at creation via the API instead of create-then-`PATCH` — see `tooling.md`. (Learned on `terraform-modules-volcano#160`.)
- Add the `autoreview` label to PRs that only touch staging environments in `kognic-cd`, `k8s-platform-gitops`, or `k8s-platform-gitops-volcano` — the autoreview bot skips unlabeled PRs. Pass `--label autoreview` to `gh pr create`, or `gh pr edit <n> --add-label autoreview` on an existing PR. Never add it to PRs that touch prod paths.
- **When posting a code review as a PR comment** (e.g. `/anthropic-review-pr:review-pr` output, or any review I ask you to post), wrap the review body in a collapsible `<details><summary>Code review results</summary>` … `</details>` block — a blank line after the `<summary>` line and before the closing `</details>` so GitHub renders the markdown inside. Keep the `` `/anthropic-review-pr:review-pr` output: `` marker line (when applicable) *above* the `<details>`, not inside it. This replaces the skill's default `> `-quote-block wrapping. Write the body with the **Write** tool (reviews routinely contain `!`, which shell redirection mangles) and post with `gh pr comment <n> --repo <owner>/<repo> --body-file <path>`.
- PR body: lead with the **intent** (why / what it achieves), not a recap of the diff. Default to **one sentence** plus any tracking link (e.g. `Part of kognic-internal/devplat#612`); the reviewer can see the changed files. Do NOT restate what the diff already shows — no listing changed files, line/occurrence counts, exact before/after strings, or step-by-step mechanism — and don't re-explain machinery documented elsewhere. Use backticks for identifiers/paths. Example (good): "Points the CIDR-subnet module uses at the first-party `modules/subnets-cidr` — now copied into terraform-modules-volcano." Add a second sentence only when a reviewer genuinely needs context that isn't visible in the diff (a non-obvious why, a dependency/ordering, a surprising gap). Only use a `## Summary` section with bullets when the change spans multiple distinct concerns that truly benefit from enumeration.
- **Always fully-qualify a cross-repo tracking-issue link to the repo the issue actually lives in** — `owner/repo#NNN` (e.g. `Part of annotell/terraform-devplat-volcano#449`, or `Part of kognic-internal/devplat#612`). A bare `#NNN`, or the *wrong* `owner/repo#NNN`, resolves to that repo's own issue/PR namespace and links to the wrong place — often a coincidental unrelated item with the same number (issues and PRs share one number space per repo, so `#NNN` may even be a PR). **Verify which repo the issue is in before linking — don't assume.** Most platform-engineering tracking issues live in `kognic-internal/devplat`, but not all (e.g. the TOS-audit work is tracked in `annotell/terraform-devplat-volcano#449`). Applies to PR bodies, commit messages, and issue/PR comments. (Learned: a bare `#449` mislinked per-repo, and a wrong-repo `kognic-internal/devplat#449` pointed at an unrelated merged PR.)
- **In PR/issue *bodies* and comments, write cross-repo references as explicit markdown links with the full URL** — `[owner/repo#NNN](https://github.com/owner/repo/pull/NNN)` — rather than relying on GitHub's `owner/repo#NNN` autolinking. A bare `#NNN` autolinks to the *current* repo's number space (wrong repo), and even the `owner/repo#NNN` text form renders inconsistently — an explicit full-URL markdown link is unambiguous and always resolves. Same for referencing a **file or doc in another repo**: link its blob URL (`https://github.com/owner/repo/blob/<branch>/<path>`), don't just inline-code the path. (Learned on `annotell/k8s-platform-gitops-volcano#606`: a cross-repo `#550` in the body resolved to the wrong repo and an ADR reference was inline code, not a link — both fixed with explicit full-URL markdown links.)
- PR body should describe what changed, not what was intentionally omitted — don't enumerate things you didn't do unless a reviewer would otherwise be surprised by the gap.
- PR body readability: short, plain sentences beat one long clause-stacked sentence. Split run-ons (especially two clauses joined by an em-dash) into separate sentences. Lead with the concrete cause a reader can picture ("every Renovate bump rewrote the generated READMEs"), not an abstract restatement ("the READMEs encode version constraints"). Prefer plain words over jargon — "reran by hand" / "recurring chore" over "manual regen" / "recurring toil", "rewrote" over "churns". (Learned on `terraform-modules-volcano#153`.)
- PR body layout: separate distinct points with **blank lines** (short paragraphs) — typically cause, then fix, then any cross-reference/tracking link. Don't cram them into one dense block, even when it's only 2-3 sentences. (Learned on `terraform-modules-volcano#162` / `terraform-devplat-volcano#455`: single-paragraph bodies read much better split into newline-separated points.)
- Never amend commits
- Never force-push. When a task would require it — e.g. rebasing a branch whose PR is already pushed — use a no-force alternative instead: `git merge --no-ff <other-branch>` brings the other branch's commits in and pushes normally.
- Never use `git -C <path>` — always `cd` to the repo directory first, then run git commands without `-C`
- `git pull --ff-only` (and bare `git fetch`) **fails on repos that move a floating major tag** — release-please action repos like `annotell/gha-golang` republish `v1`/`v2` on every release: `! [rejected] v1 -> v1 (would clobber existing tag)`. The default tag fetch refuses to overwrite the local floating tag and the command exits non-zero (the branch ref often still updates, so a later `pull` says `Already up to date` — but the non-zero exit blocks chained steps and looks like a failure). Resolve with `git fetch origin --tags --force`, then `git pull --ff-only`. `--force` only repoints the floating tag (`vN`) to its new commit; immutable release tags (`1.2.3`) are content-addressed and untouched. `--no-tags` does **not** help (the configured tag-follow refspec still clobbers). (Learned cleaning up worktrees after merging into `annotell/gha-golang`, whose release-please flow re-tags `v1`.)
- Run `git add` and `git commit` as separate Bash tool calls — never chain them with `&&` or `;`
- Never run `git commit` with `run_in_background: true` — commits are signed via 1Password, which needs an interactive prompt that backgrounded Bash can't render. The commit fails with `failed to fill whole buffer`. Always commit in the foreground.
- **Always create commits with a local, signed `git commit` — never via the GitHub API.** Local commits are SSH-signed through 1Password and show **Verified** on GitHub; commits created through the GitHub Contents API (`gh api --method PUT .../contents/...`) or any REST write are built server-side, never touch the signing key, and land **Unverified** (GitHub's web-flow signature only covers web-UI edits and merges, not user-token API writes). This means: don't script API commits to "save cloning" for multi-repo changes. Clone each target repo locally — they're usually already under `~/Code/_kognic/` (`gh repo clone annotell/<repo>` if not) — then branch/worktree, `Edit`, and `git commit` so every commit is signed. The extra clones are worth it; an Unverified commit on a shared repo is not. (Learned creating six consumer-repo PRs via the Contents API — all landed Unverified — while the two made with local `git commit` verified fine.)
- For multi-line commit messages: pass a multi-line string directly to `git commit -m` — never use command substitution (`$(...)`) in commit commands
- Match commit granularity to the task — bulk-adding new files (migrations, scaffolding) is one commit, not one per directory. Split commits only when changes are incremental to existing code and benefit from independent review or rollback.
