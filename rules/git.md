# Git

> Full war stories behind these rules: `reference/git.md` in this repo (not auto-loaded).

## Commits

- Conventional commit messages, single line: `<type>[scope]: <description>`
- Append `!` after `<type>[scope]` for a BREAKING change — but `!` means the **released artifact** breaks for its consumers (removed CLI flag, changed API), not that the change needs careful operational sequencing. CI-template flips, config changes, and rollout steps are NOT breaking; express sequencing in the PR body/merge gate. A spurious `!` forces a major version bump of an artifact that didn't change.
- Never amend commits
- Never force-push. When a task seems to require it (e.g. rebasing a pushed PR branch), use `git merge --no-ff <other-branch>` and push normally.
- Never use `git -C <path>` — always `cd` to the repo directory first
- Run `git add` and `git commit` as separate Bash tool calls — never chained (see `tooling.md`)
- **Scope `git commit` with an explicit pathspec (`git commit <paths> -m …`) whenever the repo might carry unrelated staged/uncommitted changes** — a bare `git commit` lands the *entire* index, not just what you `git add`ed. In a shared/personal notes repo mid-edit (e.g. `lindeskar/work` during a wiki cleanup), a bare commit once swept the user's in-progress work into an unrelated commit. When unsure, `git status --short` first; recover a bad bundle with `git reset --soft HEAD~1` (restores the pre-commit index, nothing lost).
- Never run `git commit` with `run_in_background: true` — 1Password signing needs an interactive prompt; backgrounded commits fail with `failed to fill whole buffer`. Same reason: a dispatched **subagent** (which runs backgrounded) can't make signed commits — in subagent-driven flows the controller makes the commits in the foreground. **In a headless/background *job* (not just `run_in_background` bash), even a foreground `git commit` can hit `failed to fill whole buffer` intermittently** — the 1Password prompt may surface for early commits then stop mid-session. For a throwaway/never-merged branch (a CI-trigger test PR), `git commit --no-gpg-sign` is an acceptable fallback; for anything that lands in history, retry once and otherwise ask the user to run the commit so the prompt appears.
- **Always create commits with a local, signed `git commit` — never via the GitHub API.** API-created commits (`gh api --method PUT .../contents/...`) land **Unverified** (they never touch the 1Password SSH signing key). Don't script API commits to "save cloning" for multi-repo changes — clone each target locally (usually already under `~/Code/_kognic/`), branch, edit, commit signed.
- For multi-line commit messages: pass a multi-line string directly to `git commit -m` — never command substitution
- Match commit granularity to the task — bulk-adding new files is one commit, not one per directory. Split only when changes are incremental and benefit from independent review or rollback.
- `git pull --ff-only` / bare `git fetch` **fails on repos that move a floating major tag** (release-please action repos like `annotell/gha-golang` republish `v1` per release): `! [rejected] v1 -> v1 (would clobber existing tag)`. Resolve with `git fetch origin --tags --force`, then pull — `--force` only repoints the floating tag; immutable release tags are untouched. `--no-tags` does not help.

## Pull requests

- Always create PRs in draft mode with a description: `gh pr create --draft --title "..." --body "..."` — no exceptions, even throwaway/test PRs (a ready PR fires CODEOWNERS review-requests and notifies colleagues before CI finishes)
- **Set the final, valid Conventional-Commit title when you *create* the PR — never placeholder-then-rename.** The `annotell` org's "Validate PR title" ruleset checks only the `opened` event and does not re-run on rename, so a placeholder's failure sticks until a new commit or reopen. For titles containing `!` (which `gh pr create --title` mangles), set the title at creation via the API from a Write-created file — see `tooling.md`.
- Add the `autoreview` label to PRs that only touch staging environments in `kognic-cd`, `k8s-platform-gitops`, or `k8s-platform-gitops-volcano` (`--label autoreview`). Never on PRs touching prod paths.
- **A throwaway change added only to trigger CI (e.g. a comment in a state dir to make a path-filtered terraform matrix run) must be reverted the moment CI confirms green — before the PR is marked ready — never left with a "remove before merge" marker.** Squash-merge folds every commit on the branch into the default branch, so the marker doesn't stop it landing; it just costs a follow-up cleanup PR. If the trigger must survive the review window, revert it in a second commit on the same branch rather than trusting anyone to strip it at merge.
- **When posting a code review as a PR comment**, wrap the body in a collapsible `<details><summary>Code review results</summary>` … `</details>` block (blank line after `<summary>` and before `</details>` so the markdown renders). Keep any `` `/anthropic-review-pr:review-pr` output: `` marker line *above* the `<details>`. Write the body with **Write** (reviews routinely contain `!`) and post with `gh pr comment <n> --repo <owner>/<repo> --body-file <path>`.

## PR/issue bodies and links

- PR body: lead with the **intent** (why / what it achieves), not a diff recap. Default to one sentence plus any tracking link; don't restate what the diff shows (no changed-file lists, counts, before/after strings). Backticks for identifiers/paths. Add a second sentence only when the reviewer needs context invisible in the diff; use a `## Summary` bullet section only for genuinely multi-concern changes.
- PR body describes what changed, not what was intentionally omitted — don't enumerate non-changes unless the gap would surprise a reviewer.
- PR body readability: short, plain sentences; split run-ons; lead with the concrete cause a reader can picture; plain words over jargon. Separate distinct points (cause → fix → cross-reference) with blank lines, even in a 2-3 sentence body.
- **Cross-repo references — one rule, three parts:** (1) always fully qualify to the repo the issue/PR actually lives in (`owner/repo#NNN`), verifying which repo that is first — a bare or wrong-repo `#NNN` resolves into the wrong number space, often hitting an unrelated item; (2) in PR/issue bodies and comments, write them as explicit full-URL markdown links (`[owner/repo#NNN](https://github.com/owner/repo/pull/NNN)`) — autolinking is repo-local and inconsistent; same for files in other repos (link the blob URL, don't inline-code the path); (3) in replies to the user, make every referenced PR/issue a clickable markdown link with the fully-qualified form as link text (`[owner/repo#NNN](https://github.com/owner/repo/pull/NNN)`), placed in the final message. **A bare or inline-code `owner/repo#NNN` in a reply does NOT autolink — it renders as dead plain text.** So the rule is absolute across all three surfaces: *never* emit a cross-repo ref as inline code or bare text anywhere (GitHub bodies/comments *and* chat replies) — always the full-URL markdown link. Recurring slip: writing `annotell/kognic-ui#889` in backticks and calling it a "link." Most plateng tracking issues live in `kognic-internal/devplat`, but not all — check.
