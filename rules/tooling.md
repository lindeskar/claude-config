# Tooling

- Python: always use `uv run` to run scripts (e.g. `uv run main.py`)
- `uv sync`/`uv run` rewrites `uv.lock` source registries to the locally-configured index (e.g. `pypi.org/simple` → `pypi.kognic.io/simple`) when the committed lock used a different index. This is local-env noise, not a real dependency change — `git checkout uv.lock` (or unstage it) before committing unless you intentionally changed deps. Check `git diff uv.lock` after any `uv` invocation in a repo you'll commit to.
- Sandbox cache redirects: many tools fail in the command sandbox when writing their default cache under `~/.cache` or `~/Library/Caches` (`Operation not permitted` / `failed to initialize cache`). These caches are now redirected globally via the `env` block in `settings.json` (`UV_CACHE_DIR`, `GOCACHE`, `XDG_CACHE_HOME`, `npm_config_cache` → fixed `/tmp/claude/*` paths, which are sandbox-writable and stable across sandboxed/non-sandboxed calls). **Do not prepend per-command cache env prefixes** for `uv`, `go`, `gh`, or `npm`/`npx` — run them bare (`go test ./...`, `gh run view …`) so the `permissions.allow` patterns match instead of falling through to the auto-mode classifier. Only add an inline cache env var for a tool not covered by the global block, and prefer a fixed `/tmp/claude/<tool>-cache` path over `$TMPDIR` (see the cross-call note below).
- Provider-plugin tools under the sandbox: `terraform`/`tofu test` (and any command that must launch the provider — `plan`, `apply`, `providers schema`) fail with `Unrecognized remote plugin message` / `Failed to read any lines from plugin's stdout` — the sandbox blocks the provider binary's go-plugin handshake. This is *not* a cache issue; rerun with the sandbox disabled. `init` works sandboxed (download only), so init in-sandbox, then run the provider-launching command with the sandbox off.
- Never `tail` (or `head`) the output of `tofu`/`terraform` `validate`/`plan` to check for errors — each error block spans ~10 lines, so `tail` silently hides earlier blocks and you'll report a partial picture. Worse, the *last* block is often the known-benign one (e.g. a `vault` `file()` token path that only resolves on the CI runner), so tailing makes a multi-error result look like "validates cleanly, only the expected error." Enumerate **all** blocks: `… 2>&1 | grep -iE 'error|warning'` to count them, then `grep -A6 'Error:'` for context. Only claim "clean" after confirming every remaining error is environment-only (missing creds/token files supplied in CI) and none reference your changed files. (Learned mis-reporting an example as "validates cleanly" when `volcengine` provider `access_key`/`secret_key` errors sat above the tailed vault error.)
- `$TMPDIR` resolves to a *different directory* in sandboxed vs non-sandboxed Bash calls (`/tmp/claude-501/...` vs `/var/folders/.../T/`). A file written to `$TMPDIR` with the sandbox off is invisible to a later sandboxed call that re-expands `$TMPDIR` (and vice versa). When a temp file must cross between calls with different sandbox settings — e.g. dump `tofu providers schema -json` sandbox-off, then `jq` it sandboxed — note the absolute path from the first call's output and use it verbatim in the next.
- Before relying on version-specific behaviour of a tool the CI runs (e.g. a `terraform`/`tofu` feature like `replace_triggered_by`), check the version the CI/runner actually pins — local and CI versions can differ and silently change behaviour. Reproduce against the CI version (download the exact binary if needed) before shipping a fix that depends on it.
- When telling the user the *exact invocation* of an unfamiliar CLI, verify its flag-passing convention (`--flag=value` vs `--flag value`, subcommand and parameter names) against `--help`, the docs, or the source before presenting it — don't assume it follows the GNU/AWS-CLI `--flag=value` norm. Many vendor CLIs don't. (Learned recommending `ve sts AssumeRole --RoleTrn=…`: the Volcengine `ve` CLI parser treats everything after `--` as the flag *name*, so `--RoleTrn=trn:…` becomes a flag literally named `RoleTrn=trn:…` and errors `… must set value`; it only accepts space-separated `--RoleTrn <value>`. Cost a troubleshooting round-trip.)
- When wiring a tool to fetch credentials dynamically (AWS `credential_process`, an `op run` chain, an assume-role helper), check the higher-precedence credential sources *first* — a static entry there silently wins and the dynamic path never runs. For AWS the order is env vars → `~/.aws/credentials` → `~/.aws/config`; a leftover static `[profile]` block in `~/.aws/credentials` shadows a `credential_process` in `~/.aws/config` → `AccessDenied` with no hint the helper was skipped. A pre-existing profile stub is a tell that partial manual setup already happened — inspect it before building on top.
- Regenerate an OpenTofu/Terraform lock file (`.terraform.lock.hcl`) without backend access: `tofu init -backend=false` installs providers + modules without remote-state creds, and `tofu providers lock -platform=… …` never touches the backend at all. Use both to regenerate a lock from a laptop that can't reach the remote state backend (e.g. an air-gapped China S3 backend): run `init -backend=false` first (installs modules, so transitive provider requirements like `hashicorp/time` are discovered), then `providers lock` listing every consuming platform — `-platform=linux_amd64` for the CI runner **plus** `-platform=darwin_arm64` for laptops (a laptop-only lock omits the runner's `h1:`). Private git module sources need the network sandbox off (registry + GitHub fetch). On macOS the `osxkeychain` git credential helper already authenticates `github.com/annotell/...`, so `init -backend=false` usually fetches the modules with **no token env at all** — try it bare first. If auth genuinely fails, add `GITHUB_TOKEN=$(gh auth token)` using the substitution form; never paste a literal resolved `gho_...` token into the command/env — the auto-mode classifier blocks that as credential leakage. When you are *changing* a pinned provider version (especially a **downgrade**), plain `init -backend=false` won't move it and `providers lock` errors (`locked provider … does not match configured version constraint; must use tofu init -upgrade`). Run `init -backend=false -upgrade` first, then `providers lock`. `-upgrade` only moves providers whose pin actually changed — sibling providers that are exact-pinned (`version = "3.7.2"`) stay put — so it's safe to run across many states for a single-provider bump/downgrade. Verify afterward that the lock diff touches only the intended provider's `version`/`hashes` (`git diff '*.terraform.lock.hcl' | grep -E '^[+-]\s*version'`).
- Generated files that CI verifies via "regenerate + `git diff --exit-code`" (terraform-docs READMEs, OpenAPI/codegen output, lock files) must be produced by the generator, never hand-written or hand-edited — a hand-made version won't match byte-for-byte and fails the check, often via an invisible diff (whole file shows changed with identical-looking content = line endings; e.g. `make docs` runs terraform-docs through docker `-t`, so its READMEs are CRLF). Run the project's generator (`make docs`, the cached docker image, etc.); if you can't, say so and leave it for the generator rather than guessing. Confirm with `file <path>` / a line-ending check when a diff looks like a phantom.
- Helm: read chart values with `helm show values`
- Use built-in tools for all file operations — never fall back to shell equivalents:
  - **Read** not `cat`, `head`, `tail`, `sed -n` — use `offset`/`limit` params to read specific line ranges
  - **Glob** not `find`, `ls`
  - **Grep** not `grep`, `rg`, `awk`
  - **Edit** not `sed`, `awk`
  - **Write** not `echo >`, `cat <<EOF`
  - Narrow exception: a single mechanical, identical edit applied across many files (e.g. a project-wide import-path or symbol rename) may use one `sed`/`perl -pi` pass when per-file `Edit` would mean dozens of calls. Only when the replacement string is identical everywhere and the result is verified afterward by a compiler/linter/test (e.g. `go build ./...`) or a `Grep` confirming zero stragglers — never for context-dependent edits.
- Never chain Bash commands with `&&`, `||`, or `;` — run each command as a separate Bash tool call. This includes `cd`, `git`, multi-step inspection (`git status` + `git diff` + `git log`), and any other composite. Reasons: the user can only approve/deny the chain as a whole, intermediate failures are hard to attribute, and an unintended state after a partial run (e.g., a worktree created from stale master) is harder to recover from. Parallel independent calls in one message are fine; chaining in one shell command is not.
- Never use command substitution (`$(...)` or backticks) in Bash tool calls — these require explicit permission every time. Instead, capture the output in a prior step and use it directly, or use built-in tools (Glob, Grep, Read) to achieve the same result. **One exception — injecting a secret**, e.g. `GITHUB_TOKEN=$(gh auth token)`: here substitution is *required*, because capturing the token and pasting the literal gets blocked by the auto-mode classifier as credential leakage. Accept the one-time prompt (see the `.terraform.lock.hcl` regen note above).
- The Bash tool runs under interactive zsh, so `!` triggers history expansion and a backslash gets inserted into the literal text — *even inside single quotes, quoted heredocs (`<<'EOF'`), and arrays*. Writing `feat!: x` to a file via `cat`/`echo` silently produces `feat\!: x` (confirm with `od -c`). This corrupts test fixtures, sample commit titles, and regexes containing `!`. To create files whose content contains `!`, use the **Write** tool, not shell redirection. When testing a regex against such data, read the data from a Write-created file rather than inlining it. This also bites CLI *arguments*, not just file content: `gh pr create/edit --title "feat!: …"` lands a literal `feat\!:` in the PR title (breaks Conventional-Commit/semver parsing). `git commit -m` happens to survive it, but the `gh` title arg does not. **Do not work around it by creating the PR with a placeholder title and renaming/`PATCH`-ing afterward** — the "Validate PR title" ruleset validates only the placeholder and the failure sticks (see `git.md` for why). Instead set the real title **at creation** via the API, reading it from a Write-created file: `gh api repos/<o>/<r>/pulls -f head=<branch> -f base=<base> -F draft=true -F title=@<titlefile> -F body=@<bodyfile>`. Verify with `gh pr view <n> --json title --jq .title | od -c`.
- Use LSP diagnostics and go-to-definition to understand and troubleshoot code — don't guess at types, imports, or call sites by grepping
- When investigating errors: check LSP/compiler output first, then read the relevant source — don't shotgun-grep the codebase
- Before asserting that a tool or system **can't** or **won't** do something — especially before writing it into long-term memory (the work wiki) or telling the user as fact — verify against a concrete artifact: a real PR, CI run, log line, or command output that shows the actual behaviour. Extrapolating a confident negative from a *related* note or general knowledge is how wrong "facts" get recorded. (Learned writing "Renovate won't regenerate `.terraform.lock.hcl`" into the wiki, extrapolated from a manual-regen memory; an existing Renovate PR — `terraform-devplat-volcano#434` — showed it updates the lock with the full multi-platform hash set. The PRs were available to check the whole time.)
- Multi-repo/multi-resource lookups via `gh api` in a shell loop hit GitHub's **secondary rate limit** after ~100+ rapid calls; the throttled calls return non-zero, so a loop that swallows errors (`… >/dev/null 2>&1`, `|| true`) silently reports every item as absent/empty — a false negative you may then assert as fact. Prefer **one batched GraphQL query** (alias per repo: `r0: repository(owner:…, name:…){…} r1: …`) when reading the same field across many repos — single request, no rate-limit risk, authoritative. If you must loop REST calls, don't discard stderr, pace them (`sleep`), and spot-check one item with a direct call before trusting an all-negative result. (Learned auditing CODEOWNERS across 58 `terraform-*` repos: a 3-paths × 56-repos loop reported "all MISSING"; a single direct call proved several already had the file — the user caught the wrong claim.)
- To copy a file **verbatim** from a GitHub repo, fetch it with `gh api repos/<owner>/<repo>/contents/<path> -H 'Accept: application/vnd.github.raw'` (redirect to a file). `WebFetch` converts the page to markdown and runs it through a summarizing model — it does **not** return byte-exact content, so it is wrong for vendoring a spec/license/template that must match upstream. After fetching, `cp` the file into place (preserving fidelity) rather than reproducing it through `Write`. (Learned vendoring the OKF `SPEC.md` into a skill's `references/`: an initial WebFetch returned a reworded summary; `gh api … raw` gave the exact file.)
- When a request says "all uses" / "everywhere" / "across the `<X>` dirs" (a plural or a glob), enumerate the full match set first (`Grep` the symbol/module path across the dir glob) and reflect the scope back — "that's N refs across M dirs, all of them?" — before editing. Don't collapse a plural/glob ("the devplat-pub dirs") to the single instance you happen to be looking at. (Learned scoping "all uses in staging dirs" to one file when the user meant every `devplat-pub-*` state dir.)
- Grafana: always use the Grafana MCP server tools for querying dashboards, logs, metrics, alerts, and incidents — never curl or WebFetch the Grafana web UI
- Grafana: when authoring PromQL/LogQL that will land in a dashboard or alert, validate it via `query_prometheus`/`query_loki_logs` first — substitute concrete sample values for `${var}` interpolation. Catches regex escaping (`\.` in double-quoted strings), `label_replace` duplicate-labelset errors, and rate-vs-counter mismatches before the dashboard ships.
- When investigating alerts, crash loops, or performance issues: invoke `kognic-devops:investigate` first — it has LogQL/PromQL patterns and datasource UIDs that prevent wasted queries
- Loki via `query_loki_logs`: apps that log structured JSON at DEBUG (e.g. renovate-ce dumps its entire resolved config per line) return single lines so large that even a `limit: 30` query blows the tool's token cap and spills to a file — and `|~` substring filters match *inside* those embedded config blobs, returning noise instead of the event you want. Pre-filter hard: exclude the dump lines with `!=` on their tell-tale keys (`!= "matchPackageNames" != "deps\":"`), narrow with **structured-metadata** equality (`| repository=\`annotell/k8s-platform-gitops\``) rather than line-substring, anchor on a distinctive event marker (`|~ "Repository (started|finished)"`, `"Creating PR"`, `"Deleted remote branch"`), and tighten the time window to minutes. To prove *which* of N replicas handled something, query all of them at once (`deployment=~"renovate-[a-d]"`) filtered to the one repo — the returned set is the answer (e.g. 327k lines scanned, 6 returned, all one shard = only that shard ran it). Going to these run logs beats theorizing about config semantics — root-caused the renovate shard PR-close race this way.
