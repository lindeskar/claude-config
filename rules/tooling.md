# Tooling

- Python: always use `uv run` to run scripts (e.g. `uv run main.py`)
- Sandbox cache redirects: many tools fail in the command sandbox when writing their default cache under `~/.cache` or `~/Library/Caches` (`Operation not permitted` / `failed to initialize cache`). Redirect the cache into `$TMPDIR` rather than disabling the sandbox â€” prepend the relevant env var to the command:
  - `uv` â†’ `UV_CACHE_DIR=$TMPDIR/uv-cache`
  - `go` (build/test) â†’ `GOCACHE=$TMPDIR/go-build`
  - `gh` (e.g. `gh run view --log-failed`) â†’ `XDG_CACHE_HOME=$TMPDIR/gh-cache`
- Provider-plugin tools under the sandbox: `terraform`/`tofu test` (and any `plan`/`apply` that must launch a provider) fail with `Unrecognized remote plugin message` / `Failed to read any lines from plugin's stdout` â€” the sandbox blocks the provider binary's go-plugin handshake. This is *not* a cache issue; rerun with the sandbox disabled. `init` works sandboxed (download only), so init in-sandbox, then run `test` with the sandbox off.
- Helm: read chart values with `helm show values`
- Use built-in tools for all file operations â€” never fall back to shell equivalents:
  - **Read** not `cat`, `head`, `tail`, `sed -n` â€” use `offset`/`limit` params to read specific line ranges
  - **Glob** not `find`, `ls`
  - **Grep** not `grep`, `rg`, `awk`
  - **Edit** not `sed`, `awk`
  - **Write** not `echo >`, `cat <<EOF`
  - Narrow exception: a single mechanical, identical edit applied across many files (e.g. a project-wide import-path or symbol rename) may use one `sed`/`perl -pi` pass when per-file `Edit` would mean dozens of calls. Only when the replacement string is identical everywhere and the result is verified afterward by a compiler/linter/test (e.g. `go build ./...`) or a `Grep` confirming zero stragglers â€” never for context-dependent edits.
- Never chain Bash commands with `&&`, `||`, or `;` â€” run each command as a separate Bash tool call. This includes `cd`, `git`, multi-step inspection (`git status` + `git diff` + `git log`), and any other composite. Reasons: the user can only approve/deny the chain as a whole, intermediate failures are hard to attribute, and an unintended state after a partial run (e.g., a worktree created from stale master) is harder to recover from. Parallel independent calls in one message are fine; chaining in one shell command is not.
- Never use command substitution (`$(...)` or backticks) in Bash tool calls â€” these require explicit permission every time. Instead, capture the output in a prior step and use it directly, or use built-in tools (Glob, Grep, Read) to achieve the same result.
- Use LSP diagnostics and go-to-definition to understand and troubleshoot code â€” don't guess at types, imports, or call sites by grepping
- When investigating errors: check LSP/compiler output first, then read the relevant source â€” don't shotgun-grep the codebase
- Grafana: always use the Grafana MCP server tools for querying dashboards, logs, metrics, alerts, and incidents â€” never curl or WebFetch the Grafana web UI
- Grafana: when authoring PromQL/LogQL that will land in a dashboard or alert, validate it via `query_prometheus`/`query_loki_logs` first â€” substitute concrete sample values for `${var}` interpolation. Catches regex escaping (`\.` in double-quoted strings), `label_replace` duplicate-labelset errors, and rate-vs-counter mismatches before the dashboard ships.
- When investigating alerts, crash loops, or performance issues: invoke `kognic-devops:investigate` first â€” it has LogQL/PromQL patterns and datasource UIDs that prevent wasted queries
